## 一、Token：模型的"字母"

### 1.1 什么是 Token？

**一句话理解**：Token 是模型处理文本的最小单位

**类比**：
```
人类阅读：字/词
模型处理：Token

中文例子：
文本: "机器学习"
Token: ["机器", "学习"]  或  ["机", "器", "学", "习"]

英文例子：
文本: "unhappiness"
Token: ["un", "happiness"]  或  ["un", "happ", "iness"]
```

### 1.2 为什么需要 Tokenization？

**原因 1：统一表示**
- 不同语言有不同字符集
- Token 将所有文本转为数字 ID

**原因 2：控制词表大小**
- 英文单词 > 100万，太多
- Token 词表通常 3万-10万，可控

**原因 3：处理未登录词**
- "ChatGPT4Turbo" 可能不在词表
- 切分为 ["Chat", "G", "PT", "4", "Turbo"] 就能处理

### 1.3 Token 与成本的直接关系

```
OpenAI API 定价（每 1M tokens）：
┌──────────────────┬──────────────────┬──────────────────┐
│      模型         │     输入价格      │     输出价格      │
├──────────────────┼──────────────────┼──────────────────┤
│ GPT-4o           │ $2.50            │ $10.00           │
│ GPT-4o mini      │ $0.15            │ $0.60            │
│ GPT-3.5 Turbo    │ $0.50            │ $1.50            │
└──────────────────┴──────────────────┴──────────────────┘

实际例子：
一篇 1000 字的中文文章 ≈ 1500 tokens
调用 GPT-4o 总结一次 ≈ $0.004
```

### 1.4 Token 的上下文限制

```
模型                上下文长度      实际意义
─────────────────────────────────────────────────────
GPT-3.5             16K            约 12,000 汉字
GPT-4               128K           约 100,000 汉字
Claude 3            200K           约 150,000 汉字
Gemini 1.5          1M-2M          约 1 本书

重要提示：
- 上下文不是越长越好，模型可能"遗忘"前面的内容
- 关键信息放在 Prompt 的开头或结尾效果更好
```

## 二、Embedding：语义的空间表示

### 2.1 核心概念

**一句话理解**：Embedding 是把文字变成「语义坐标」的过程

**类比**：
```
现实世界：地址 → GPS 坐标
          "北京市朝阳区" → (116.4, 39.9)

语义世界：词语 → 向量坐标
          "机器学习" → [0.23, -0.56, 0.89, ...] (768维)
```

### 2.2 为什么 Embedding 有用？

**特性 1：语义相近 = 向量相近**
```
文本A: "我喜欢苹果"
文本B: "我爱吃苹果"
文本C: "我讨厌香蕉"

向量距离: A-B 很近，A-C 很远
          ↓
        语义相似度
```

**特性 2：语义运算**
```
经典例子：
"国王" - "男人" + "女人" ≈ "女王"
"北京" - "中国" + "日本" ≈ "东京"

原理：向量运算捕捉了语义关系
```

### 2.3 Embedding 维度选择

| 维度 | 特点 | 适用场景 |
|------|------|----------|
| 384 | 轻量、快速 | 实时检索、移动端 |
| 768/1024 | 平衡 | 通用场景（推荐） |
| 1536/3072 | 高精度 | 复杂语义匹配 |

### 2.4 Embedding 模型选型

**决策树**：
```
主要语言？
├── 中文为主 → BGE-large-zh（开源）或 OpenAI text-embedding-3
├── 英文为主 → OpenAI text-embedding-3-large 或 E5
└── 多语言 → BGE-m3 或 OpenAI

预算限制？
├── 充足 → OpenAI（省心，效果好）
└── 紧张 → BGE（开源，可本地部署）

数据敏感？
├── 是 → 本地部署开源模型
└── 否 → 调用 API
```

## 三、Temperature：创造力的旋钮

### 3.1 直观理解

**一句话理解**：Temperature 控制模型输出的「随机性」

**类比**：
```
Temperature = 0.0   → 机器人，确定性强
Temperature = 0.7   → 人类，平衡
Temperature = 1.0+  → 艺术家，创意发散
```

### 3.2 实际效果对比

```
Prompt: "给咖啡店起个名字"

Temperature = 0.2:
- 星巴克咖啡
- 上岛咖啡
- 太平洋咖啡
（安全、常规）

Temperature = 0.7:
- 晨光咖啡
- 醇香小馆
- 慢时光
（合理、有新意）

Temperature = 1.5:
- 量子泡沫咖啡
- 月亮与六便士
- 咖啡因宇宙
（天马行空，可能不相关）
```

### 3.3 场景推荐

| 场景 | 推荐 Temperature | 原因 |
|------|-----------------|------|
| 代码生成 | 0.0 - 0.3 | 确定性高，减少 bug |
| 数据分析 | 0.0 - 0.3 | 需要准确结果 |
| 问答/客服 | 0.3 - 0.5 | 准确为主，适度自然 |
| 写作/创意 | 0.7 - 1.0 | 需要多样性 |
| 头脑风暴 | 1.0 - 1.5 | 追求创意发散 |

### 3.4 Top-p 与 Top-k

**Top-k**：只从概率最高的 k 个词中选择
```
k=1: 总是选最可能的词（贪婪解码）
k=50: 从高概率的 50 个词中采样
```

**Top-p（核采样）**：从累积概率达到 p 的最小集合中选择
```
p=0.9: 只考虑累积概率 90% 的词
更动态，适应不同分布
```

**推荐组合**：
```
精确场景：Temperature=0.2, Top-p=0.1
平衡场景：Temperature=0.7, Top-p=0.9
创意场景：Temperature=1.0, Top-p=0.95
```

## 四、幻觉（Hallucination）：AI 的"脑补"

### 4.1 什么是幻觉？

**定义**：模型生成看似合理但实际错误或无根据的内容

**类型 1：事实性幻觉**
```
用户：谁发明了电话？
模型：托马斯·爱迪生发明了电话。
事实：亚历山大·贝尔发明了电话。
```

**类型 2：忠实性幻觉**
```
文档："该产品价格为 100 元"
用户：产品价格是多少？
模型："该产品售价为 200 元"
问题：与原文矛盾
```

### 4.2 为什么会产生幻觉？

**原因 1：训练目标导致的**
```
LLM 的训练目标：生成"听起来合理"的文本
                    ↓
              不是追求"事实正确"
                    ↓
              可能编造细节使文本流畅
```

**原因 2：知识截止**
```
模型训练后就"冻结"了
不知道之后发生的事件
可能用旧信息"脑补"
```

**原因 3：上下文限制**
```
长文档处理时可能"遗忘"关键信息
或者混淆不同段落的内容
```

### 4.3 如何减少幻觉？

**策略 1：RAG（检索增强）**
```
给模型提供可靠的信息源
限制在提供的资料内回答
```

**策略 2：提示工程**
```
Prompt: "请基于以下文档回答，如果无法找到答案请说'不知道'：
[文档内容]

问题：..."
```

**策略 3：后处理验证**
```
- 事实核查（搜索验证）
- 一致性检查（多次采样对比）
- 置信度校准（让模型评估自己的不确定性）
```

**策略 4：模型选择**
```
- 更强的模型幻觉率更低
- 使用专门微调的"诚实"模型
- 考虑使用多模型投票
```

## 五、上下文学习（In-Context Learning）

### 5.1 核心概念

**一句话理解**：模型能从 prompt 中的示例学习新任务，无需修改参数

**传统机器学习**：
```
训练阶段：大量数据 → 调整模型参数 → 得到模型
推理阶段：新输入 → 模型预测

需要：重新训练（耗时、昂贵）
```

**上下文学习**：
```
Prompt: 
"翻译以下句子：
英文: Hello → 中文: 你好
英文: World → 中文: 世界  
英文: [新句子] → 中文:"

模型自动理解：这是翻译任务，遵循前面的模式
```

### 5.2 三种学习形式

| 形式 | 示例数量 | 适用场景 |
|------|---------|----------|
| **Zero-shot** | 0 | 简单任务，模型已熟悉 |
| **One-shot** | 1 | 快速展示格式 |
| **Few-shot** | 2-10 | 复杂任务，需要示范 |

### 5.3 Few-shot 设计技巧

**技巧 1：示例多样性**
```
❌ 不好的示例：
"苹果" → 水果
"香蕉" → 水果
"橙子" → 水果

✅ 好的示例：
"苹果" → 水果
"苹果" → 公司（上下文：科技公司）
"跑步" → 运动
```

**技巧 2：示例顺序**
```
- 把最相关的示例放在后面（离问题更近）
- 或者按难度递增排列
```

**技巧 3：示例格式**
```
保持一致格式，包括：
- 分隔符
- 标签名称
- 输出格式
```

### 5.4 思维链（Chain of Thought, CoT）

**核心发现**：让模型"一步步想"，推理能力大幅提升

**对比**：
```
普通 Prompt:
Q: 一个农场有鸡和兔，头共 35 个，脚共 94 只，各多少？
A: 鸡 23 只，兔 12 只。
（直接给答案，容易错）

CoT Prompt:
Q: 一个农场有鸡和兔...
A: 让我一步步思考：
   1. 假设全是鸡，35×2=70 只脚
   2. 实际多 94-70=24 只脚
   3. 每只兔多 2 只脚，24÷2=12 只兔
   4. 鸡有 35-12=23 只
   答案：鸡 23 只，兔 12 只。
（分步推理，正确率高）
```

**CoT 触发方式**：
1. **Few-shot CoT**：prompt 中提供推理示例
2. **Zero-shot CoT**：加一句"Let's think step by step"

## 六、RLHF：用人类偏好训练模型

### 6.1 为什么需要 RLHF？

**问题**：预训练模型能生成流畅文本，但：
- 可能生成有害内容
- 可能不遵循指令
- 可能产生偏见

**目标**：让模型输出符合人类价值观和偏好

### 6.2 RLHF 三步骤

```
步骤 1：收集偏好数据
┌─────────────────────────────────────────────────────────────┐
│  问题: "如何制作炸弹？"                                      │
│                                                             │
│  回答 A: [详细制作步骤]   ← 有害                            │
│  回答 B: "我不能提供这种信息"  ← 安全                       │
│                                                             │
│  人类标注: B > A                                            │
└─────────────────────────────────────────────────────────────┘

步骤 2：训练奖励模型 (Reward Model)
┌─────────────────────────────────────────────────────────────┐
│  输入: 问题 + 回答                                           │
│  输出: 分数（代表人类喜好程度）                               │
│                                                             │
│  训练目标: 排序损失（好回答分数 > 差回答分数）               │
└─────────────────────────────────────────────────────────────┘

步骤 3：强化学习优化 (PPO)
┌─────────────────────────────────────────────────────────────┐
│  目标: 让策略模型生成高奖励的回复                             │
│                                                             │
│  约束: 不要偏离原始模型太远（KL 散度惩罚）                   │
│                                                             │
│  算法: PPO (Proximal Policy Optimization)                  │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 RLHF 的效果

| 方面 | 改进 |
|------|------|
| **安全性** | 减少有害输出 |
| **有用性** | 更好地遵循指令 |
| **诚实性** | 更愿意承认不知道 |
| **无害性** | 避免生成偏见内容 |

### 6.4 RLHF 的局限

- **主观性**：不同人偏好不同
- **对齐税**：过度对齐可能降低模型能力
- **可扩展性**：需要大量人工标注

**新方向**：RLAIF（AI Feedback），用 AI 代替人类进行偏好判断

## 七、量化：模型瘦身技术

### 7.1 什么是量化？

**一句话理解**：用更少的位数存储模型参数，减少显存占用

**类比**：
```
原图（高清）：每个像素 32 位浮点数
压缩图：每个像素 8 位整数（肉眼难辨差别）

模型（FP32）：每个参数 32 位
量化（INT8）：每个参数 8 位（体积减 75%）
量化（INT4）：每个参数 4 位（体积减 87.5%）
```

### 7.2 量化精度对比

| 格式 | 每参数位数 | 7B模型大小 | 精度损失 | 适用场景 |
|------|-----------|-----------|---------|----------|
| FP32 | 32 | 28 GB | 0% | 训练 |
| FP16/BF16 | 16 | 14 GB | <1% | 推理 |
| INT8 | 8 | 7 GB | 1-2% | 通用推理 |
| INT4 | 4 | 3.5 GB | 2-5% | 资源受限 |

### 7.3 量化方法

**GPTQ**：逐层量化，适合 GPU
- 一次量化，永久使用
- 推理速度快

**GGUF**：llama.cpp 格式，CPU/GPU 通吃
- 多种量化级别可选（Q4_K_M, Q5_K_M 等）
- 适合本地运行

**AWQ**：保护重要权重，精度更好
- 识别并保留关键参数
- 相同大小下效果更好

### 7.4 显存需求速查

```
计算公式：显存(GB) ≈ 参数数量(B) × 精度(位) ÷ 8

7B 模型：
- FP16: 14 GB
- INT8: 7 GB  
- INT4: 3.5 GB（实际约 4-5GB 含开销）

13B 模型：
- FP16: 26 GB
- INT4: 6.5 GB

70B 模型：
- FP16: 140 GB（需多卡）
- INT4: 35 GB（单卡 A100 可跑）
```

| 概念 | 一句话解释 | 关键数字 |
